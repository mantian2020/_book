Channel的register的本质

本质就是将channel注册到线程NioEventLoop中的Selector上，让Selector帮我们监听感兴趣的IO事件(accept,read,write,connect)。如图所示：

Channel的register流程

Channel的register流程分为 启动时Channel的register流程 和 新连接Channel的register流程 两大部分内容。

启动时Channel的register流程

研究server和client端的register流程代码入口分别为：ServerBootstrap#bind方法和Bootstrap#connect方法。无论server端还是client端入，它们最终都会调到AbstractBootstrap#initAndRegister方法，我们直接看这个方法。

    Channel channel = null;

        ......

    ChannelFuture regFuture = config().group().register(channel);

从config().group().register(channel);这行代码开始register流程，这个register方法调用的就是NioEventLoopGroup父类MultithreadEventLoopGroup的register方法。

// MultithreadEventLoopGroup 类
public ChannelFuture register(Channel channel) {
    return next().register(channel);
}
// next()方法
public EventLoop next() {
    return (EventLoop) super.next();
}

其中的next()方法是通过选择器chooser选择一个线程NioEventLoop，调用该线程的register方法，把channel注册到上去，让我们来看看该线程的register方法，其实调用的是父类SingleThreadEventLoop的register方法。

// SingleThreadEventLoop类
public ChannelFuture register(Channel channel) {
    return register(new DefaultChannelPromise(channel, this));
}
public ChannelFuture register(final ChannelPromise promise) {
    ObjectUtil.checkNotNull(promise, "promise");
    // 调用AbstractChannel的register方法，当前线程NioEventLoop作为参数，传递过去了
    promise.channel().unsafe().register(this, promise);
    return promise;
}

在这篇文章《Netty源码-Channel讲解》中讲到创建channel时，ACCEPT/READ事件以参数形式传递给父类AbstractNioChannel的readInterestOp成员变量，什么时候把ACCEPT/READ事件绑定到Selector上呢？不再展开细说了，最终会在AbstractNioChannel的doBeginRead方法中绑定到Selector上，有兴趣的同学，可以按照下面的流程打断点跟踪下。

AbstractBootstrap#doBind0-->....-->DefaultChannelPipeline#bind-->...

--> DefaultChannelPipeline.HeadContext#bind

--> AbstractChannel#bind(){...pipeline.fireChannelActive()...}

--> DefaultChannelPipeline.HeadContext#channelActive(){...readIfIsAutoRead()...}--> ....

---> DefaultChannelPipeline.HeadContext#read

--> AbstractNioChannel#doBeginRead(){...selectionKey.interestOps(interestOps | readInterestOp);}

Channel的register流程源码先分析到这，本篇只研究了启动时Channel的register流程，还剩下新连接Channel的register流程没有分析，等研究完线程EventLoop后，在续上剩下的部分。

总结

  • register的本质是channel注册到selector上，而不是EventLoop上，selector帮忙监听ACCEPT/READ事件。
  • 线程EventLoop创建时，没有立即运行，而是在第一个任务提交时运行的。
